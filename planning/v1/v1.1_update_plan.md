# ego-memory-anchor 架构升级计划 (v1.1)

本计划基于 `project_review.md` 中提出的潜在风险，制定针对工程落地期间的技术痛点的应对和优化路线。所有的调整需后续合并入 `DESIGN.md` 与 `TASKS.md`。

## 阶段 1：构建与存储优化 (Storage & Media)
**目标**：解决大文件（尤其是视频高达 500MB）产生的加载卡顿、Egress 带宽过高及浏览器内存溢出问题。

1. **富媒体预处理流 (前端端侧控制)**：
   * 在 `UploadZone.tsx` 内直接引入端侧轻量化压缩组件或视频转码方案（例如 `@ffmpeg/ffmpeg` wasm 版本，视性能定）。
   * 对图片统一在端侧校验，如果检测到 `.heic`，自动引入 `heic2any` wasm 库在端侧无感转换为 `image/jpeg`，保证全平台（尤其非 Mac 生态）的可看性。
2. **渐进式数据处理**：
   * 为应对微信超大 ZIP 文件解析导致的浏览器崩溃（OOM）：重写 `lib/utils/wechat-parser.ts`，将其拆解为基于 Web Worker 的异步后台计算流。
   * 在产品页面上增加“按年份分批上传”的用户引导，约束单次上传 ZIP 大小不超限。

## 阶段 2：鉴权与性能重构 (DB Performance)
**目标**：修复 `memories` 列表在高并发/大并发查询时的 RLS (Row Level Security) N+1 性能降级。

1. **索引优化 (Database Indexes)**：
   * 更新 `supabase/migrations/` 下的相关脚手架配置，强制追加 `family_members(profile_id, user_id)` 的联合索引，从而提升现行 RLS 中 EXISTS 语句的匹配效率。
2. **未来规划评估 (Custom Claims)**：
   * 对超过万人或单档超过万条记忆的请求路径，平替 RLS 长链。计划把用户的角色 (Role/Profile Auth) 在登录阶段由 edge compute 签发到 JWT custom claims 内。

## 阶段 3：生命周期与合规 (Lifecycle)
**目标**：账号注销时的资源真级联清理。

1. **废弃文件批量收集 (Orphan File Cleaner)**：
   * 新增 Supabase Edge Function: `delete-user-data`。
   * 配置 Database Webhook/Trigger：当 `auth.users` 发起注销事件，不仅从 Postgres 里删除关系链，同时在后台异步列举其全部 `Storage Items` 并发起完全擦除。
