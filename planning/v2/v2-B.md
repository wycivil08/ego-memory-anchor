# 文档 1/4：PRODUCT\_V2.md — 差异化产品设计

```markdown
# PRODUCT_V2.md — 忆锚 vs 大公司网盘：精细差异化设计

## 0. 核心命题

> 用户的替代方案不是 iCloud 或百度网盘——是"什么都不做，让记忆慢慢消散"。
> 忆锚要赢的不是存储之战，而是意义之战。

---

## 1. 专属档案（Memorial Profile）到底是什么

### 1.1 概念定义

专属档案不是"文件夹"。它是一个 **为逝者而存在的数字纪念空间**——
有名字、有面容、有日期、有关系、有家人共建、有文化节奏。

| 维度 | iCloud / 百度网盘 | 忆锚的专属档案 |
|------|------------------|-------------|
| 基本单元 | 文件夹 | **有身份的纪念空间**（姓名、头像、生卒日期、关系） |
| 组织方式 | 用户手动分类 | **自动时间线**（按 EXIF/手动日期生成生命叙事） |
| 权限模型 | "分享链接"或"共享文件夹" | **家庭角色制**（owner/editor/viewer，邀请制） |
| 情感属性 | 无 | **纪念日系统**（生日、忌日、清明，含农历） |
| 真实性保障 | 无 | **"原始记录"徽章**（不可移除，每条素材自动标注） |
| 叙事功能 | 无 | **家人注释**（多人视角，共同构建故事） |
| 文化适配 | 无 | **农历日历 + 清明提醒 + 文化节日整合** |
| 打开仪式感 | 打开一个工具 | **打开一个人的生命故事** |

### 1.2 专属档案的核心交互设计

**进入仪式**：用户从 Dashboard 点击一张逝者的卡片 → 进入的不是文件列表，
而是一条从上到下的 **生命时间线**——最早的记忆在底部，最近的在顶部。
页面顶端是逝者的头像、姓名、生卒日期，和家人写的一句话描述。
这个"进入"动作本身就是一次微型的纪念仪式——
类似于走进一间存放老照片的房间，而不是打开一个 U 盘。

**Dashboard 呈现**：

```

┌──────────────────────────────────────────────────┐
│  忆锚 · 我守护的记忆                  \[用户头像 ▼]│
├──────────────────────────────────────────────────┤
│                                                  │
│  ┌─────────────┐  ┌─────────────┐               │
│  │   \[头像]     │  │   \[头像]     │               │
│  │   爷爷       │  │   小橘       │               │
│  │ 1945 - 2023 │  │ 2016 - 2024 │               │
│  │   327 条记忆 │  │   89 条记忆  │               │
│  │ "永远笑眯眯" │  │ "最爱晒太阳" │               │
│  └─────────────┘  └─────────────┘               │
│                                                  │
│  ┌─────────────┐                                │
│  │     +        │  后天是爷爷的生日               │
│  │  为 TA 建一  │  ← 纪念日温馨提示横幅           │
│  │  个记忆空间  │                                │
│  └─────────────┘                                │
│                                                  │

```

每张卡片不是冰冷的文件夹图标，而是 **逝者的头像 + 姓名 + 关系 + 素材数量 + 一句话描述**。
视觉上它像一个 "人"，不像一个 "容器"。

### 1.3 专属档案内的时间线设计

**时间线不是文件列表，是生命叙事。**

```

```
**关键设计细节：**

| 元素 | 设计意图 | 心理学依据 |
|------|---------|-----------|
| **"原始记录"徽章** | 每条素材自动标注，不可编辑/删除。传递"这是真实的" | Anchoring 理论：锚定感来自不可篡改的证据 |
| **注释 + 注释者署名** | 不同家人对同一素材的记忆/故事可能不同，多视角构建叙事 | Meaning-making：整合丧失进人生叙事是主动心理工作 |
| **按年→月→日分组** | 时间线 = 生命叙事，而非文件列表 | Identity reconstruction：看到完整一生有助于重建自我认知 |
| **"日期未知"组** | 允许模糊，不逼迫用户确定所有日期 | 降低摩擦 — 用户认知资源极度有限 |
| **语音内联播放** | 不需要跳转就能听到真实的声音 | 研究反复证明"听到真实的声音"是最具情感冲击力的特性 |
| **家人贡献者头像** | 看到"这条是弟弟上传的" | Social validation：纪念是家庭集体行为 |

### 1.4 "原始记录"信任架构

这是忆锚与所有竞品/网盘的 **最深层差异化**。

**设计规则：**

1. 平台上的每一条素材都自动获得 **"原始记录"** 标记
2. 这个标记在 UI 上始终可见（每条卡片、详情页）
3. 标记不可被用户编辑或删除
4. 数据库中 `source_label` 字段 DEFAULT '原始记录'，无 UPDATE 入口
5. 如果未来引入任何 AI 辅助（如旧照片修复），修复后的版本 **必须** 标注 "AI 增强"，且原件永远保留

**视觉实现：**

```

┌────────────────────────────┐
│ \[照片缩略图]                │
│                            │
│ 2023-10-01                 │
│ 📷 照片 · \[原始记录]        │  ← 小徽章，stone-200 背景 + stone-600 文字
│                            │      不抢眼但始终在场
│ "国庆节全家最后一次聚餐"     │
└────────────────────────────┘

```

**心理学意义：** Julia Shaw 的研究警告 "AI 是完美的虚假记忆机器"。
我们的产品通过在视觉层面 **持续确认真实性**，为用户提供心理安全感——
"你看到的、听到的、读到的，都是真实发生过的"。
这是大公司网盘做不到的，因为它们没有"真实性"的概念。

### 1.5 纪念日系统 — 将产品嵌入文化仪式

| 网盘 | 忆锚 |
|------|------|
| 无日历概念 | **自动识别生日、忌日** |
| 无文化适配 | **支持农历日期（清明、忌日常用农历）** |
| 无主动触达 | **温馨提醒："后天是爷爷的生日"** |
| 无仪式引导 | **提醒日附带一条随机记忆** |

**提醒横幅设计：**

```

┌────────────────────────────────────────────┐
│ 🕯️ 后天是爷爷的生日（农历二月初三）          │
│    "永远记得他笑着说'来来来，吃饺子'"        │  ← 随机展示一条该档案的注释
│                                    \[查看 →] │
└────────────────────────────────────────────┘

```

这让产品从"工具"变成"仪式的一部分"——
清明节前用户自然会想起来打开忆锚，不是因为 push 通知，
而是因为产品已经成为了他们纪念实践的组成部分。
这正是研究中提到的"嵌入现有文化纪念实践"的设计原则。

### 1.6 家庭共建 — 增长引擎 + 心理学正确

**家庭共建不是"分享功能"——它是产品的核心增长引擎和心理学基石。**

| 网盘共享 | 忆锚家庭共建 |
|---------|------------|
| "我发给你一个文件" | "我们一起建设这个纪念空间" |
| 单向传递 | 多向贡献 |
| 无角色概念 | owner / editor / viewer |
| 无归属感 | "4 位家人共同守护" |

**家庭共建的心理学依据：**

- **Social validation 需求**：Tony Walter 的研究表明，communal recognition 是哀伤处理的必要条件
- **Meaning-making 是集体行为**：不同家庭成员对同一个人的记忆不同，汇集后构成更完整的叙事
- **家庭裂变 = 自然增长**：一个用户创建档案 → 邀请 2-5 个家人 → 部分家人为另一位逝者创建档案 → 再次邀请

**邀请流程设计（极低摩擦）：**

```

1. Owner 点击 "邀请家人"
2. 选择角色：查看 / 编辑
3. 生成链接（无需填写邮箱 — 微信友好）
4. 复制链接 → 微信/短信发给家人
5. 家人点击链接 → 看到逝者信息 + "加入此记忆空间"
6. 注册/登录后自动关联

```

**关键：不要求输入邮箱。** 中国用户的第一通信工具是微信，
"复制链接发到微信群"比"输入对方邮箱"的转化率高 10 倍。
```

```
### 2.3 V2 新增的差异化功能设计

#### 功能 A：微信语音消息专项导入

**为什么单独设计：** 微信语音消息是中国用户最珍贵、也最容易丢失的记忆素材。
微信语音有 180 天过期机制（聊天文件自动清理），且不支持直接转发/保存。
"妈妈发来的最后一条语音"可能在手机清理时永远消失——这就是"第二重丧失"。

**MVP 实现路径：**
- 教用户手动保存语音文件（微信"收藏" → "以文件方式发送"）
- 提供清晰的图文教程
- 上传后自动识别为语音类型
- 时间线内可直接播放（不需要点进详情页）

#### 功能 B：记忆空间封面 + 一句话描述

**不只是"备注"——它是家人的集体记忆浓缩。**

每个专属档案除了头像外，还有：
- **一句话描述**（如 "永远笑眯眯的老爷子"）— 任何 editor 都可以编辑
- **封面照片**（可选）— 从已上传照片中选一张作为档案封面

这些元素让"打开档案"的瞬间就有情感浓度，不是看到一个空壳。

#### 功能 C：导出 = 信任

**导出功能不藏在设置页深处。它出现在：**
1. 设置页面 — 最醒目的位置
2. 每个档案的"编辑"页面 — 明确标注"你随时可以带走全部数据"
3. Landing Page — 隐私承诺区块写明"一键导出，永不锁定"

**导出包结构：**
```

爷爷的记忆\_2026-04-15.zip
├── 照片/
│   ├── 2023-10-01\_国庆聚餐.jpg
│   ├── 2022-06-15\_父亲节.jpg
│   └── ...
├── 视频/
├── 语音/
├── 文字记录/
├── 扫描件/
├── metadata.json          ← 所有素材的元数据+注释+标签
└── README.txt             ← "这是你从忆锚导出的全部数据，
格式通用，不依赖任何平台。"

````

# 文档 2/4：MARKETING.md — 冷启动与营销策略

```markdown
# MARKETING.md — 忆锚冷启动与增长策略

## 1. 创始人故事框架

### 1.1 为什么创始人故事是最强信任杠杆

研究（文档4）反复印证：grief tech 领域最成功的案例都有一个创始人亲身经历的故事。
- Silicon Intelligence 孙凯 — 为母亲做 AI 头像
- HereAfter AI 创始人 — 因外祖父去世创业
- Joshua Barbeau — 用 GPT 重建已故未婚妻（虽非创始人，但媒体传播力极强）

**小公司的信任不来自品牌，来自人格。**

### 1.2 故事模板（供创始人填写）

````

\[段落 1：触发]
**年\_\_\_\_月，我失去了**。
（描述丧失的具体情境——突然的还是有预期的？当时的感受是什么？）

\[段落 2：第二重丧失]
几个月后，我发现了一件比失去 TA 更让我害怕的事：
关于 TA 的记忆正在消散。
（具体描述：换了手机、某段微信语音过期了、找不到某张照片了、
记忆开始模糊——TA 到底是怎么笑的？声音到底是什么样的？）

\[段落 3：行动]
我开始把所有能找到的照片、视频、语音消息收集到一起。
翻遍了旧手机、微信聊天、云盘、家人的相册……
这个过程很痛苦，但每找到一条记录，我都觉得——
"还好，这段记忆还在。"

\[段落 4：产品动机]
我做忆锚，是因为我不希望任何人经历这种"第二重丧失"。
手机会换代，账号会注销，云盘会过期——
但关于 TA 的真实记忆，不应该因为技术原因而消散。

\[段落 5：承诺]
忆锚不"复活"任何人，不生成 AI 对话，不合成任何内容。
我们只做一件事：守护你已有的真实记录，让它们永远不丢失。


````
## 2. 冷启动路径（0 → 50 种子用户）

### 2.1 种子用户画像

**不是"任何失去亲人的人"——是"正在主动寻找解决方案的人"。**

| 特征 | 描述 |
|------|------|
| 年龄 | 25-45 岁 |
| 触发时间 | 丧亲 1 天 ~ 2 年内 |
| 行为特征 | **主动在小红书/知乎搜索** "怎么保存去世亲人的照片""微信语音怎么永久保存""怎么纪念去世的宠物" |
| 技术素养 | 能独立操作手机和电脑，能按教程导出微信聊天记录 |
| 情感状态 | 不是急性期崩溃，而是过了最初的冲击后开始"想做些什么" |

### 2.2 启动阶段（第 1-4 周）
````

````
## 2. 关键合规项详解

### 2.1 ICP 备案

**要求：** 在中国大陆提供互联网信息服务，必须完成 ICP 备案。

**忆锚的情况：**
- 如果服务器在中国大陆（如阿里云/腾讯云）→ 必须 ICP 备案
- 如果服务器在海外（Vercel/Supabase）且仅面向海外用户 → 不需要备案，
  但中国大陆用户访问速度慢且可能被 DNS 干扰

**操作步骤：**
1. 注册企业主体（个体工商户/有限公司均可，建议有限公司）
2. 在云服务商（如阿里云）的备案系统提交材料
3. 等待管局审批（通常 5-15 个工作日）
4. 备案通过后在网站底部展示备案号

**MVP 建议：** 先用海外部署面向海外华人验证需求（无需备案）。
同时启动企业注册 + ICP 备案流程，备案通过后迁移到国内服务器。

### 2.2 个人信息保护法 (PIPL) — 核心合规项

**忆锚涉及的敏感个人信息：**

| 数据类型 | PIPL 分类 | 敏感等级 |
|---------|---------|---------|
| 用户邮箱/手机号 | 个人信息 | 一般 |
| 逝者照片 | 个人信息（可能含人脸） | 敏感 |
| 逝者语音 | 个人生物识别信息 | **敏感** |
| 微信聊天记录 | 个人信息 + 通信内容 | **敏感** |
| 家庭关系信息 | 个人信息 | 一般 |
| 逝者生卒日期 | 个人信息 | 一般 |

**PIPL 要求与忆锚的应对：**

| PIPL 要求 | 忆锚应对措施 | MVP 实现 |
|---------|------------|---------|
| **明示同意** — 收集敏感信息需单独同意 | 注册时弹出隐私政策 + 上传照片/语音时二次确认 | ✅ |
| **最小必要** — 只收集必需数据 | 产品不收集任何非用户主动上传的数据 | ✅ 已满足 |
| **删除权** — 用户可要求删除全部数据 | "删除账号"功能 → 级联删除所有数据 | ✅ |
| **数据可携带权** — 用户可导出数据 | "数据导出"功能 → ZIP 下载 | ✅ |
| **个人信息保护影响评估** | 处理敏感信息前应做评估并留档 | ⚠️ 需补充 |
| **个人信息保护负责人** | 企业需指定负责人 | ⚠️ 需在企业注册后指定 |
| **告知义务** — 向用户说明处理目的/方式/范围 | 隐私政策详细说明 | ✅ |

**关于逝者的数据保护：**
PIPL 第 49 条规定："自然人死亡的，其近亲属为了自身的合法、正当利益，
可以对死者的相关个人信息行使本章规定的查阅、复制、更正、删除等权利。"
→ 忆锚的使用场景（近亲属整理逝者记忆）天然符合这一条款。
但需在隐私政策中说明上传者保证自己是逝者的近亲属或获得近亲属授权。

### 2.3 数据跨境传输 — MVP 最大风险点

**如果使用 Vercel（美国）+ Supabase（海外）：**

PIPL 第 38 条要求：向境外提供个人信息须满足以下条件之一：
1. 通过国家网信部门组织的安全评估
2. 经专业机构进行个人信息保护认证
3. 按照标准合同与境外接收方订立合同
4. 法律法规规定的其他条件

**实操路径：**

| 方案 | 描述 | 适用场景 |
|------|------|---------|
| **方案 A：海外部署，仅面向海外华人** | 不触及数据跨境问题 | MVP 验证阶段 |
| **方案 B：国内云部署** | 数据不出境，完全合规 | 面向国内用户 |
| **方案 C：标准合同备案** | 与 Supabase/Vercel 签标准合同并向网信办备案 | 中小规模跨境 |

**MVP 建议：先走方案 A → 验证需求 → 迁移到方案 B。**

方案 B 的国内替代方案：

| 海外服务 | 国内替代 | 差异 |
|---------|---------|------|
| Vercel | 腾讯云 Webify / 阿里云 FC | 需适配部署流程 |
| Supabase (Postgres) | 腾讯云 PostgreSQL / 阿里云 PolarDB | API 不同需迁移 |
| Supabase Storage | 腾讯云 COS / 阿里云 OSS | SDK 不同需迁移 |
| Supabase Auth | 自建 Auth / AuthJS | 需重写认证层 |

**迁移成本预估：2-4 周开发工作。** 建议在 MVP 架构中做好抽象层，
便于后续切换底层服务。

### 2.4 网络安全等级保护（等保）

**等保二级适用于：** 信息系统遭到破坏后，会对公民、法人和其他组织的合法权益
产生严重损害，或者对社会秩序和公共利益造成损害。

**忆锚的情况：** 存储大量用户私密照片和语音，泄露会对用户造成严重损害。
但 MVP 阶段用户量极小，实操中不会被追究。

**建议时间线：**
- 用户 < 1,000：不做等保，但做好安全基础（加密、RLS、备份）
- 用户 1,000-10,000：启动等保二级测评
- 用户 > 10,000：必须完成等保二级
````

```markdown
## 2. GDPR 合规分析（最严格，满足它基本满足全球）

### 2.1 合法性基础 (Legal Basis)

GDPR 要求每一项数据处理都有合法性基础。忆锚的情况：

| 数据处理活动 | 合法性基础 | 理由 |
|-------------|---------|------|
| 收集邮箱/密码 | **合同必要** (Art.6(1)(b)) | 提供服务的必要条件 |
| 存储用户上传的照片/视频/语音 | **合同必要** + **明确同意** | 产品核心功能 + 涉及敏感数据需额外同意 |
| 处理逝者的个人数据 | **合法利益** (Art.6(1)(f)) | GDPR Recital 27 明确指出 GDPR 不适用于已故人士数据，但各成员国可自行立法。部分国家（如法国）有逝者数据保护法 |
| 发送纪念日提醒邮件 | **明确同意** | 用户主动设置提醒 |

### 2.2 数据主体权利对照

| GDPR 权利 | 条款 | 忆锚实现 | 状态 |
|-----------|------|---------|------|
| **知情权** | Art.13/14 | 隐私政策详细说明 | ✅ |
| **访问权** | Art.15 | 用户可随时查看所有数据 | ✅ |
| **更正权** | Art.16 | 用户可编辑/删除任何素材和注释 | ✅ |
| **删除权（被遗忘权）** | Art.17 | 一键删除账号 + 全部数据 | ✅ |
| **数据可移植性** | Art.20 | 一键导出 ZIP（通用格式） | ✅ |
| **限制处理权** | Art.18 | 可冻结档案（暂停处理） | ⚠️ 需补充 |
| **反对权** | Art.21 | 无自动化决策/画像，不适用 | ✅ N/A |

### 2.3 GDPR 特殊要求

| 要求 | 说明 | 忆锚应对 | MVP 必须 |
|------|------|---------|---------|
| **DPO（数据保护官）** | 大规模处理敏感数据时需指定 | MVP 规模小，暂不需要 | ❌ |
| **DPIA（数据保护影响评估）** | 高风险处理活动需提前评估 | 存储私密照片/语音属高风险，应做 | ⚠️ 建议做 |
| **数据处理记录** | Art.30 — 维护处理活动记录 | 创建简单的处理活动登记表 | ⚠️ 建议做 |
| **数据泄露通知** | Art.33/34 — 72 小时内通知 | 建立泄露响应流程 | ⚠️ 建议做 |
| **跨境传输** | Chapter V | Vercel/Supabase 在美国 — 需 SCCs 或 adequacy decision | ⚠️ 需关注 |
| **Cookie 同意** | ePrivacy Directive | 忆锚不用追踪 cookie，仅功能性 session cookie | ✅ |

### 2.4 关于逝者数据的欧盟特殊规定

**GDPR 本身不适用于已故人士**（Recital 27），但部分成员国有额外规定：

| 国家 | 法规 | 内容 |
|------|------|------|
| **法国** | Loi Informatique et Libertés Art.85 | 数据主体可在生前对死后数据处理留下指示；近亲属可行使已故者的数据权利 |
| **意大利** | GDPR 实施法令 Art.2-terdecies | 近亲属可代行访问/更正/删除权 |
| **西班牙** | LOPDGDD Art.3 | 继承人/亲属可行使已故者的数字权利 |
| **德国** | BGH 判例法 | 数字账户是遗产的一部分，继承人有权访问 |

**忆锚的应对：** 在用户协议中明确：上传者保证自己是逝者的近亲属或遗产继承人，
有权处理逝者的个人数据。

## 3. 美国 CCPA/CPRA 合规分析

### 3.1 适用范围

CCPA/CPRA 适用于在加州开展业务且满足以下任一条件的企业：
- 年收入 > $25M
- 购买/出售/分享 > 100,000 消费者数据
- 50%+ 收入来自出售消费者数据

**忆锚 MVP 阶段很可能不触及门槛。** 但建议从开始就按 CCPA 标准设计。

| 选择退出出售/共享 | 不出售不共享 ✅ N/A |
| 更正权 | 可编辑所有内容 ✅ |
| 限制使用敏感信息 | 不做任何敏感信息的二次利用 ✅ |

研究文档（文档4）中提到 Craig Klugman 等学者呼吁将 griefbots 归类为医疗器械，

## 4. 日韩合规概要

### 韩国 PIPA
- 跨境传输需获得数据主体同意或确认接收国保护水平
- 忆锚应在隐私政策中说明数据存储地点并获取同意
- 韩国对"数字遗产"有较活跃的讨论但尚未立法

### 日本 APPI
- 跨境传输需确认对方国有等同保护水平，或获得数据主体同意
- 日本有强大的"先祖崇拜"文化传统，数字纪念产品接受度较高
- 无特定悲伤科技法规
```
